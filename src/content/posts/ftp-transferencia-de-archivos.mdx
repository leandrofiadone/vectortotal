HTTP y FTP mueven datos por TCP. Hasta ahí el parecido. HTTP usa una conexión para todo. FTP abre dos: una para los comandos, otra para los datos. Esa separación resuelve un problema de eficiencia — y crea uno de exposición. La contraseña viaja por el canal de control, en texto plano, antes de que se transfiera un solo archivo.

## Dos conexiones

| Conexión | Puerto | Función |
|----------|--------|---------|
| **Control** | 21 | Comandos (`USER`, `LIST`, `RETR`…) |
| **Datos** | 20 (activo) o dinámico (pasivo) | Archivo o listado de directorio |

Cuando ejecutás `ls`, el cliente manda `LIST` por el canal de control y el servidor abre **una nueva conexión** para enviar el listado. Lo mismo pasa con cada descarga — cada transferencia es una conexión nueva.

**Modo activo** — el servidor inicia la conexión de datos hacia el cliente. Es la excepción a la regla habitual: acá es el servidor el que se conecta, no el cliente. Por eso falla con firewalls — el cliente tiene que aceptar una conexión entrante que no inició.

**Modo pasivo (PASV)** — el cliente inicia ambas conexiones. El servidor responde con un puerto donde conectarse. Es el modo estándar hoy.

```
227 Entering Passive Mode (10,10,41,192,134,10).
```

Los últimos dos números codifican el puerto: `134 × 256 + 10 = 34314`.

## Comandos del protocolo

Lo que escribís en el cliente FTP y lo que realmente viaja por la red son cosas distintas. La interfaz es amigable. El protocolo es texto crudo:

| Lo que escribís | Lo que viaja | Función |
|-----------------|-------------|---------|
| (login) | `USER anonymous` | Usuario |
| (password) | `PASS` | Contraseña |
| `ls` | `LIST` | Listar directorio |
| `get archivo` | `RETR archivo` | Descargar |
| `put archivo` | `STOR archivo` | Subir |
| `type ascii` | `TYPE A` | Modo texto |
| `quit` | `QUIT` | Cerrar sesión |

<Callout type="info">
  **Login anónimo**: muchos servidores FTP públicos aceptan el usuario `anonymous` con cualquier contraseña (o vacía). Es acceso público deliberado, no un error.
</Callout>

## Una sesión real

El canal de control de FTP es completamente legible en una captura. Esto es lo que aparece — login, listado, descarga:

<Step n={1} title="Conexión y login">
```
$ ftp 10.66.181.12
Connected to 10.66.181.12.
220 (vsFTPd 3.0.5)
Name: anonymous
331 Please specify the password.
Password:
230 Login successful.
```
</Step>

<Step n={2} title="Listar y preparar">
```
ftp> ls
227 Entering Passive Mode (10,10,41,192,134,10).
150 Here comes the directory listing.
-rw-r--r--  1 0  0  1480 Jun 27 08:03 coffee.txt
-rw-r--r--  1 0  0    14 Jun 27 08:04 flag.txt
-rw-r--r--  1 0  0  1595 Jun 27 08:05 tea.txt
226 Directory send OK.
ftp> type ascii
200 Switching to ASCII mode.
```
</Step>

<Step n={3} title="Descargar" last={true}>
```
ftp> get coffee.txt
227 Entering Passive Mode (10,10,41,192,57,100).
150 Opening BINARY mode data connection for coffee.txt (1480 bytes).
226 Transfer complete.
1480 bytes received in 8e-05 secs (18500.00 Kbytes/sec)
ftp> quit
221 Goodbye.
```

Dos `PASV` distintos: uno para el listado, otro para la descarga. Cada operación de datos abre y cierra su propia conexión.
</Step>

<Callout type="warning">
  Toda la sesión — incluyendo usuario y contraseña — viaja en texto plano. Cualquiera en la misma red que capture el tráfico puede leer las credenciales completas.
</Callout>

## Referencia rápida

```bash
ftp <IP>               # conectar
ls                     # listar directorio remoto
get archivo.txt        # descargar
put archivo.txt        # subir
type ascii             # modo texto
type binary            # modo binario (default)
quit                   # salir

# Filtros Wireshark
ftp                    # canal de control
ftp-data               # transferencia de datos
```

---

FTP mueve archivos. Pero las organizaciones no solo mueven archivos — se comunican. Y para eso hay otra capa de protocolos, igual de expuesta, que maneja el envío, la recepción y la sincronización del correo.

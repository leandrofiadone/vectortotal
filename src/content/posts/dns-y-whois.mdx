Antes de que tu browser envíe un solo byte, ya salió un paquete de tu máquina que no pediste explícitamente. Una consulta DNS. Viajó, volvió con una IP, y recién ahí arrancó todo lo demás. Ese mecanismo invisible es el primero que vale la pena entender — porque es el primero que ocurre en cualquier conexión.

## Cómo funciona la resolución

Escribís `example.com` y el sistema ejecuta este proceso en milisegundos:

<Step n={1} title="Consulta al resolver local">
  Tu sistema operativo pregunta al **resolver DNS local** (generalmente el router o `127.0.0.53`). Si ya resolvió ese dominio recientemente, devuelve la respuesta desde caché — sin salir a la red.
</Step>

<Step n={2} title="El resolver consulta hacia arriba">
  Si no está en caché, el resolver sube la cadena: resolver → servidor raíz → servidor TLD (`.com`, `.org`) → servidor autoritativo del dominio. Cada eslabón sabe a quién preguntarle al siguiente.
</Step>

<Step n={3} title="Respuesta y caché" last={true}>
  El servidor autoritativo devuelve el registro pedido. El resolver lo guarda en caché según el **TTL** del registro y se lo entrega a tu sistema. La próxima consulta al mismo dominio sale desde caché.
</Step>

¿Y si el resolver responde con una IP que no es la real? Eso es DNS spoofing — alguien en el camino intercepta la consulta y devuelve lo que quiere antes de que llegue la respuesta legítima. Por ahora asumí que el resolver es confiable. No siempre lo es.

## Los tipos de registros DNS

Los registros DNS son el mapa público de cómo está organizado un dominio. Cada tipo expone algo diferente:

| Registro | Qué hace | Ejemplo |
|----------|----------|---------|
| **A** | Mapea hostname a IPv4 | `example.com → 93.184.215.14` |
| **AAAA** | Mapea hostname a IPv6 (quad-A) | `example.com → 2606:2800:...` |
| **CNAME** | Alias — apunta un dominio a otro dominio | `www.example.com → example.com` |
| **MX** | Servidor de correo del dominio | Necesario para recibir emails |

<Callout type="info">
  Cuando escribís `example.com` en el browser, se consulta el registro **A**. Cuando tu cliente de correo manda un email a `user@example.com`, consulta el registro **MX**. Mismo dominio, consultas distintas — cada tipo de registro revela una parte diferente de la infraestructura.
</Callout>

## Resolución en la práctica: nslookup

Con `nslookup` podés hacer esas consultas directamente, sin pasar por el browser:

```bash
nslookup www.example.com
```

```
Server:         127.0.0.53
Address:        127.0.0.53#53

Non-authoritative answer:
Name:   www.example.com
Address: 93.184.215.14
Name:   www.example.com
Address: 2606:2800:21f:cb07:6820:80da:af6b:8b2c
```

**"Non-authoritative"** significa que la respuesta vino del caché de un intermediario, no del servidor que administra el dominio. Es la respuesta que obtiene casi siempre cualquier cliente — la autoritativa solo aparece si le preguntás directamente al servidor responsable del dominio.

```bash
nslookup -type=MX example.com       # servidores de correo
nslookup -type=AAAA example.com     # solo IPv6
nslookup example.com 8.8.8.8        # usar el DNS de Google
```

## Qué pasa a nivel de paquetes

Una sola consulta `nslookup www.example.com` genera cuatro paquetes — el cliente consulta A y AAAA en paralelo:

```bash
tshark -r dns-query.pcapng -Nn
```

```
1  0.000000  192.168.66.89 → 192.168.66.1  DNS  Standard query  A    www.example.com
2  0.059049  192.168.66.1  → 192.168.66.89 DNS  Standard query response  A  93.184.215.14
3  0.059721  192.168.66.89 → 192.168.66.1  DNS  Standard query  AAAA www.example.com
4  0.101568  192.168.66.1  → 192.168.66.89 DNS  Standard query response  AAAA 2606:2800:21f:...
```

- Paquetes **1 y 3** — queries del cliente (A e IPv6 en paralelo)
- Paquetes **2 y 4** — respuestas del servidor
- Todo por UDP puerto 53, resuelto en menos de **110 ms**

Esto ocurre antes de que se abra una sola conexión TCP. El tráfico DNS de una red es un registro continuo de adónde intenta conectarse cada dispositivo.

## WHOIS: quién registró ese dominio

DNS te dice *a dónde va* un dominio. WHOIS responde *quién lo controla*.

Registrar un dominio exige dejar datos de contacto. Esa información es pública por diseño del sistema — no por descuido del dueño:

```bash
whois ejemplo.com
```

```
Domain Name: EJEMPLO.COM
Creation Date: 1993-04-02T00:00:00Z
Registrar Registration Expiration Date: 2026-10-20T14:56:17Z
Registrar: GoDaddy.com, LLC

Registrant Name: Registration Private
Registrant Organization: Domains By Proxy, LLC
```

Lo que podés extraer:

- **`Creation Date`** — cuándo fue registrado el dominio
- **`Expiration Date`** — cuándo vence (útil para detectar dominios abandonados)
- **`Registrar`** — empresa que gestiona el registro
- **`Registrant`** — quién lo registró (nombre, org, email, teléfono)

<Callout type="info">
  Muchos dominios usan **privacy protection** (como "Domains By Proxy" en el ejemplo). Los datos del dueño se reemplazan por los del servicio de privacidad. El nombre queda oculto — pero los registros DNS, el historial de cambios y la infraestructura asociada siguen siendo públicos.
</Callout>

## WHOIS en reconocimiento

WHOIS no es solo datos administrativos. En la fase de reconocimiento de un pentest o investigación OSINT, es una de las primeras fuentes porque el sistema fue diseñado para ser consultable por cualquiera:

- Verificar si un dominio pertenece realmente a la organización objetivo
- Encontrar emails del registrante para correlacionar con otras fuentes
- Detectar dominios próximos a vencer que podrían ser registrados por un tercero
- Correlacionar otros dominios registrados bajo la misma entidad

```bash
whois objetivo.com               # WHOIS de dominio
whois -h whois.arin.net 8.8.8.8  # WHOIS de una IP
```

<Callout type="warning">
  El WHOIS de IPs (ARIN en Norteamérica, RIPE en Europa, LACNIC en Latinoamérica) te dice a qué organización pertenece un bloque de direcciones — útil para distinguir si una IP es de AWS, Cloudflare, o de la propia empresa objetivo.
</Callout>

## Comandos de referencia

```bash
# DNS — consultas básicas
nslookup ejemplo.com                 # registro A
nslookup -type=MX ejemplo.com        # servidores de correo
nslookup ejemplo.com 8.8.8.8         # forzar DNS de Google

# dig — alternativa más detallada
dig ejemplo.com                      # registro A con TTL y flags
dig MX ejemplo.com                   # registro MX
dig AAAA ejemplo.com +short          # solo la IPv6, sin decoración

# WHOIS
whois ejemplo.com                    # info de dominio
whois 1.1.1.1                        # info de IP (Cloudflare)

# Captura de tráfico DNS
tcpdump -i eth0 port 53 -n           # tiempo real
tshark -r captura.pcap -Y dns        # filtrar DNS en un archivo

# Filtro en Wireshark
dns
```

---

Con DNS sabés a qué IP apunta un nombre. Con WHOIS, quién está detrás. Es el punto de partida — el contorno del target antes de hacer ninguna conexión. Lo que sigue es diferente: en lugar de preguntar sobre el target, vas a hablarle directamente.

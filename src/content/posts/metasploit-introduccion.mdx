En el [post de CVE-2024-21413](/salas/cve/cve-2024-21413-moniker-link) usamos un script Python manual para explotar la vulnerabilidad: buscar un PoC en GitHub, adaptarlo, debuggearlo, correrlo. Funciona, pero es artesanal. Cada vulnerabilidad nueva requiere empezar de cero.

Metasploit es la respuesta a ese proceso. En lugar de armar todo a mano, tenés un framework con **miles de módulos listos para usar** — buscás, configurás, ejecutás. Es la diferencia entre fabricar una herramienta cada vez que necesitás una y abrir una navaja suiza.

## Qué es Metasploit

El framework de explotación más usado en pentesting. Cubre todas las fases: reconocimiento, scanning, explotación, post-explotación.

## Los tres conceptos base

Antes de hablar de módulos, tres términos que van a aparecer en absolutamente todo:

```
Vulnerabilidad → la puerta rota
Exploit        → la ganzúa que la abre
Payload        → lo que hacés una vez adentro
```

**Vulnerabilidad** — el error en sí. Un bug en el código, un diseño malo, una lógica rota. Existe independientemente de si alguien la explota.

**Exploit** — el código que aprovecha esa vulnerabilidad. Es el "cómo entro". El `exploit.py` del post de [MonikerLink](/salas/cve/cve-2024-21413-moniker-link) era exactamente esto — un script que armaba el email malicioso para triggear el bug.

**Payload** — lo que querés que pase después de entrar. El exploit abre la puerta, el payload es lo que hacés adentro. Puede ser abrir una [shell](/salas/linux-fundamentals/shells-de-linux), crear un usuario, ejecutar un comando, instalar un backdoor.

## Los 7 tipos de módulos

Metasploit organiza todo en siete categorías. Es una taxonomía que vale entender porque define cómo pensás el ataque.

### Auxiliary — reconocimiento y escaneo

Todo lo que no es explotar directamente. Scanners, crawlers, fuzzers, sniffers. Es el reconocimiento — antes de atacar, necesitás saber qué hay en la red.

Subcarpetas relevantes: `scanner` (la más usada), `dos`, `sqli`, `spoof`, `gather`.

### Exploits — los módulos de ataque

Los módulos que efectivamente comprometen la máquina. Organizados por sistema operativo: `windows`, `linux`, `android`, `apple_ios`, `osx`, `multi`. También hay carpetas para sistemas legacy como `aix`, `solaris`, `mainframe` — infraestructura vieja que todavía existe en grandes empresas.

### Payloads — el código que corre en la víctima

Tiene cuatro subcategorías:

<Compare
  headers={["Qué hace", "Cómo lo identificás en el nombre"]}
  rows={[
    ["Singles", "Payload completo en un solo paquete, autosuficiente", "shell_reverse_tcp (con _)"],
    ["Stagers", "Payload chico que establece conexión y baja el stage", "shell/reverse_tcp (con /)"],
    ["Stages", "Payload pesado que descarga el stager", "Se combina automáticamente con el stager"],
    ["Adapters", "Envuelve un payload en otro formato (ej: PowerShell)", "Prefijo del formato destino"],
  ]}
/>

La forma de distinguir **staged vs single** en el nombre es clave:

```
generic/shell_reverse_tcp        ← _ entre shell y reverse = SINGLE (inline)
windows/x64/shell/reverse_tcp   ← / entre shell y reverse = STAGED
```

<Callout type="tip" title="¿Cuándo usar staged vs single?">
  **Single** cuando el canal es estable y no hay restricciones de tamaño. **Staged** cuando el canal inicial es limitado (ej: un buffer overflow con espacio reducido) — el stager es chico, pasa por el hueco, y después baja el payload completo.
</Callout>

### Encoders — evasión básica de antivirus

Codifican el payload para intentar evadir antivirus. Los AV tradicionales comparan archivos contra una base de datos de firmas conocidas — si el payload está codificado diferente, puede no matchear. Tienen éxito limitado porque los antivirus modernos también analizan comportamiento. Organizados por arquitectura: `x86`, `x64`, `php`, `cmd`.

### Evasion — evasión avanzada

Más sofisticado que los encoders. No solo cambia cómo se ve el payload, cambia cómo se comporta para evadir detección activa. Históricamente solo para Windows (AppLocker, Windows Defender).

### NOPs — relleno técnico

Instrucciones que le dicen al CPU "no hagas nada este ciclo". Se usan como relleno para que los payloads tengan tamaños exactos y consistentes. Metasploit lo maneja automáticamente — no es algo con lo que vas a interactuar directamente.

### Post — después de comprometer

Módulos para después de entrar. Escalar privilegios, moverse lateralmente por la red, extraer credenciales, mantener persistencia. Organizados por plataforma: `windows`, `linux`, `android`, `osx`, `multi`.

## msfconsole — la terminal de Metasploit

Se lanza con `msfconsole`. Funciona como una terminal Linux normal — podés hacer `ls`, `ping`, `clear` — pero con contexto propio.

### Los cinco prompts que vas a ver

Importante no confundirlos — cada uno habilita comandos distintos:

```
root@kali:~#                              → terminal Linux normal
msf6 >                                    → Metasploit abierto, sin módulo
msf6 exploit(windows/smb/ms17_010...) >  → dentro de un módulo
meterpreter >                             → explotaste la máquina, estás adentro
C:\Windows\system32>                      → shell directa en la víctima
```

<Callout type="warning" title="Contexto importa">
  Si ejecutás `set RHOSTS 10.10.1.1` en el prompt `msf6 >` sin haber seleccionado un módulo, no pasa nada útil. Si ejecutás `sessions` dentro de un módulo, funciona. Prestale atención al prompt para saber dónde estás parado.
</Callout>

## EternalBlue — el ejemplo histórico

Vale conocer la historia porque es uno de los exploits más famosos de la historia de la ciberseguridad — y el que vas a usar para practicar.

### La vulnerabilidad

Un error matemático en **SMBv1** — el protocolo de Windows para compartir archivos en red (el mismo protocolo que vimos en el post de [MonikerLink](/salas/cve/cve-2024-21413-moniker-link) filtrando hashes NTLM). Al restar un número de 32 bits de uno de 16 bits, se produce un **buffer overflow** en el kernel que permite ejecución de código arbitrario.

### El origen

Supuestamente desarrollado por la **NSA** como arma cibernética. En abril de 2017 el grupo **Shadow Brokers** filtró públicamente el exploit junto con otras herramientas de la NSA.

Un mes después, en mayo de 2017, el ransomware **WannaCry** lo usó para infectar más de 200.000 máquinas en 150 países en cuestión de horas. Hospitales, empresas, gobiernos — todo cayó.

<Mermaid title="La cadena de EternalBlue a WannaCry" chart={`
flowchart LR
  NSA["NSA desarrolla\nEternalBlue"] --> SB["Shadow Brokers\nlo filtra (abril 2017)"]
  SB --> WC["WannaCry lo usa\n(mayo 2017)"]
  WC --> IMPACT["200.000+ máquinas\n150 países"]
`} />

### Por qué sigue importando

Windows 7 y Server 2008 siguen corriendo en muchas redes corporativas. En CTFs y pentesting real, EternalBlue aparece constantemente. Es el módulo con el que vas a practicar en los próximos posts.

## Lo que viene

Este post cubre los fundamentos — la taxonomía de módulos, los comandos básicos y el contexto histórico. La serie de Metasploit continúa con:

- **Metasploit Exploitation** — usar módulos para comprometer máquinas reales
- **Meterpreter** — el payload más poderoso de Metasploit y todo lo que permite hacer en post-explotación
- **Blue** — una máquina Windows real para hackear aplicando todo lo aprendido, con EternalBlue como protagonista

## Conexiones con el blog

- El `exploit.py` manual de [CVE-2024-21413](/salas/cve/cve-2024-21413-moniker-link) es exactamente lo que Metasploit reemplaza con módulos listos para usar
- EternalBlue explota **SMB** — el mismo protocolo que filtra hashes NTLM en MonikerLink
- Los hashes que extraés en post-explotación se crackean con [John the Ripper](/salas/crypto/john-the-ripper) (`--format=nt` para NTLM local, `--format=netntlmv2` para capturas de red)
- `msfconsole` es una [shell](/salas/linux-fundamentals/shells-de-linux) especializada — funciona como Bash pero con contexto de explotación
- Reverse shells y payloads conectan directamente con el concepto de shells interactivas y no interactivas
